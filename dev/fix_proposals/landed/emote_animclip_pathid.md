# Emote AnimationClip path_id derivation — blocked

## Symptom

Object-identity-aligned bit_diff_atlas (c5e5a66) surfaces a top-residual signal
on the glb-emote bundle `bafybeickh2wiibpue…` (~849k ppm): both
`cascade_from_extra_object.AnimationClip` (248M bits) and
`cascade_from_missing_object.AnimationClip` (215M bits). Diagnostic: ours emits
1 AnimationClip with one PathID, prod emits 1 with a different PathID — same
m_Name, same `m_AnimationType`, same Animator + AnimatorController PathIDs —
so the structural diff is JUST the clip's PathID.

## What we have today

`builder.rs:1373` for the emote mecanim path:
```
let recycle = self.unique_recycle("animations", &name);
let pid = self.add("AnimationClip", clip,
                   Role::Glb("AnimationClip".into(), recycle));
```

Resolved via `Role::Glb`:
```
lid = xxh64("Type:AnimationClip->animations/<clip_name>0", seed=0)
pid = prefab_packed_path_id(glb_guid, lid, FILE_TYPE_META_ASSET=3)
```

For the `super_emote` case this yields exactly the observed `ours` pid
`3292058881938926566`, so the derivation we're running is well-understood; it
is the WRONG one.

## What we've ruled out

Brute-force probes (no Unity disassembly) over:

1. **All Unity main-asset local_ids** {2100000, 2800000, 4900000, 7400000,
 9100000, 11400000, 21300000} × **8 file types** (2 + 3) × **~300 plausible
 guid input strings** (variants of `{root_hash}/{component}/{clip_name}`
 where `component ∈ {Animator, AnimatorController, animatorController,
 Animation, animations, clip, clips, AnimationClip,...}`, with/without
 `.anim` suffix, with/without root_hash prefix, with all separators).
2. **Sub-asset of AC**: `prefab_packed_path_id(ac_guid, lid, ft)` for
 `lid ∈ {0..10_000_000} ∪ {-10M..0}` for both ft. Zero matches.
3. **Sub-asset of GLB**: `prefab_packed_path_id(glb_guid, lid, ft)` for
 `lid ∈ {0..50_000_000}` for both ft. Zero matches.
4. **Recycle-name variants** with `Role::Glb` style derivation: every
 permutation of `{short_type ∈ {AnimationClip, AnimationClip_mecanim,
 AnimationClipMecanim, Animation, Clip, MecanimClip}, recycle ∈ many}` ×
 both file_types. Zero matches.
5. **CRC32 / XXH64 of clip name** as direct local_id with both `ac_guid` and
 `glb_guid`. Zero matches.
6. **MD4 path_id** (the `path_id` function as opposed to
 `prefab_packed_path_id`). Zero matches.

The prod `-8089024512132275905` is generated by `prefab_packed_path_id` (the
function signature is forced by how Unity's SerializedFile path-IDs are
constructed), but the (guid_input, local_id, file_type) triple feeding it
is none of the obvious ones.

## Where the answer lives (and why we can't read it)

`asset-bundle-converter/Assets/AssetBundleConverter/AssetBundleConverter.cs:566`
shows the emote-mecanim path:

```csharp
var filePath = $"{animatorRoot}animatorController.controller";
var controller = AnimatorController.CreateAnimatorControllerAtPath(filePath);
…
foreach (AnimationClip animationClip in clips) {
    var newCopy = Object.Instantiate(animationClip);
    newCopy.name = animationClip.name;
    AssetDatabase.AddObjectToAsset(newCopy, controller);  // ← here
    …
    controller.AddMotion(newCopy, 0);  // 1st call
    …
    controller.AddMotion(newCopy, 0);  // 2nd call (loop-back)
}
```

The `AssetDatabase.AddObjectToAsset(newCopy, controller)` call asks **Unity
itself** to assign the clip a local file ID within the `.controller` asset.
Unity's algorithm for this assignment is internal — the LocalFileIdentifier
math for named sub-assets uses an FNV1a-ish hash of `(name, classID,
parentLfid?, importer-class?)` that has been observed to be stable per Unity
version but is not publicly specified. The `.controller` YAML records the
assigned local file ID on disk; the bundle build then propagates it into the
PathID via `prefab_packed_path_id(parent_meta_guid, lfid, file_type)`.

This is the same class of problem as the `m_TOS` ordering blocker documented
in `dev/fix_proposals/animator_controller_tos.md` (which is the AC's
hash-set iteration order): a non-public Unity algorithm. The no-disassembly
rule blocks both.

## What would close it (when the disassembly rule lifts, or a probe exists)

Option A — **probe Unity directly**. Add an `AbgenBundleProbe.cs` IPC method
that takes a controller-with-clips and dumps each clip's `.controller`-YAML
`fileID` next to its `m_Name`. Run for the 2 corpus emotes; reverse-engineer
the (name → lfid) mapping; reproduce it in `pathids.rs`. The
`AbgenBundleProbe::ProbeAcTOS` method added in prior parallel-agent work is the right
template.

Option B — **emit a Unity importer-side override**. Convince the Decentraland
converter to assign a deterministic local file ID for each clip when calling
`AddObjectToAsset` (e.g. via the `localIdentfierInFile` field, which Unity's
ScriptedImporter exposes). Then mirror the same formula here. Requires
upstream cooperation.

Option C — **structural diff acceptance**. The bit_diff_atlas's new
identity-aligned attribution surfaces this as 463M bits/2-bundle-sample but
the actual impact is 2 emote bundles in the 2,174-bundle windows corpus
(~0.09% of bundles). On the bigger picture this is a known-blocked tag, not
a real parity regression; the next bit_diff_atlas run with the alignment fix
will tag these as `cascade_from_{extra,missing}_object.AnimationClip` exactly
so the bookkeeping reflects "structural emission gap, blocked", not
"per-byte content drift, fixable".
