# Draco normals: production decodes through the 1.4.1 integer-abs bug

**Why it matters:** Every draco-compressed glb (`KHR_draco_mesh_compression`) with a NORMAL attribute diverged from the reference on its entire normal lane — and, because tangents are recomputed from normals, the tangent lane too. This was previously believed to be an inherent "merged-mesh weld + native plugin" wall. It is not a weld and not noise: it is one line of arithmetic, and it is exactly reproducible.

**The mechanism:** The converter decodes draco through DracoUnity 4.0.2, whose native libraries are built from draco 1.4.1. In that version, `OctahedronToolBox::OctahedralCoordsToUnitVector` computes the derived octahedral component as `1.f - abs(y) - abs(z)` with the C integer `abs()` in scope — the floats are truncated toward zero before the magnitude is taken. Since `|y|, |z| < 1` for every interior grid point, `x` evaluates to exactly `1.0f`, the diamond fold for the back hemisphere never triggers, and the decoded vector is `normalize(1, y, z)` instead of the octahedral mapping. Draco 1.5.x fixed the call to `std::abs`, which silently changes nearly every decoded normal — our vendored 1.5.7 produced the *correct* vectors, the reference bundles carry the *buggy* ones.

**How it was proven:** On the shuriken mesh (3907 vertices), positions and indices were byte-identical while normals diverged up to 125 degrees. Our normals sit exactly on the octahedral dequantization grid (`scale = 2/(2^q - 2)`, q=10) — pure 1.5.7 decode plus the glTF→Unity x-negation. Enumerating every quantized `(s,t)` pair, mapping each of our stored normals back to its integers, and re-decoding through the 1.4.1 truncated form predicts the reference normal **byte-exactly for 3907/3907 vertices** (accounting for the many-to-one fold collisions in the inverse map). After patching, the bundle's normal lane is byte-identical and the residual collapsed from ~45 KB of diff to six 1-ULP tangent bytes (the known tangent rounding class).

**The fix:** `third_party/draco_decoder/.../normal_compression_utils.h` keeps the 1.4.1-faithful form `1.f - std::trunc(std::abs(y)) - std::trunc(std::abs(z))`, with a comment carrying the provenance. The integer decode pipeline (prediction schemes, canonicalization) is untouched — only the final int→float dequantization changes, and that function's single caller is the normal attribute inverse transform. Positions, texcoords, and connectivity are unaffected.

**If the converter ever upgrades DracoUnity** to a package built from draco >= 1.5, production normals will jump to the correct octahedral decode and this patch must be reverted to plain `std::abs`.
