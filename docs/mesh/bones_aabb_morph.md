# Morph-aware per-bone AABB recompute

**Why it matters:** A mesh's `m_BonesAABB` holds one axis-aligned bounding box per bind-pose bone, in bone-local space, used at runtime as a conservative bound for skinned-mesh culling. For meshes that carry blendshapes (morph targets), Unity's importer recomputes these boxes after folding morph deltas into the rest pose, so each box covers every blendshape configuration the bone can reach. Computing the boxes from the base pose alone undershoots Unity's values and prevents byte-identical mesh output.

**How it works:** For each vertex and each bone it is bound to, abgen-rs contributes both the base position and the morphed position (base plus each target's delta) to that bone's box, taking the union. Meshes without morph targets fall back to the base-only loop and are unaffected. This closes the gap for most affected bones but is not a complete reproduction: a residual remains where Unity's per-bone bounds extend further than either the base or fully-morphed vertex positions can explain, and where bounds appear to draw on vertices not bound to the bone in question. That residual is believed to originate in Unity's FBX-ingest path (avatar/skeleton import), which may emit per-bone bounds hints or add virtual vertices before recomputing bounds; it is not derivable from the glTF source alone.
