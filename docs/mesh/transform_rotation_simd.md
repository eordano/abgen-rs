# Per-lane quaternion normalization for `m_LocalRotation`

**Why it matters:** Transform `m_LocalRotation` values were diverging from the reference on the `y` and `w` lanes by a single ULP, while `x` and `z` matched. Because a Transform's CAB hash depends on its serialized TypeTree, every off-by-one rotation bit also shifts the Transform's PathID, which in turn salts the PathIDs of dependent objects (notably MeshRenderer), cascading the divergence far beyond the rotation field itself.

**How it works:** The reference does not normalize the quaternion with a single global scale. Its `Transform::SetLocalRotation` computes two different sums-of-squares with different inner pair-grouping and applies each to a different pair of lanes: the `x` and `z` lanes divide by `sqrt((x²+y²)+(z²+w²))`, while the `y` and `w` lanes divide by `sqrt((x²+w²)+(y²+z²))`. The math is f32 throughout; only the inner addition order differs between the two lane-pairs, producing different round-off on each. This is the observable fingerprint of a SIMD horizontal-add inside that normalize (an `_mm_hadd_ps`/`_mm_dp_ps`-style shuffle where the even-index and odd-index lanes carry different round-off paths). Earlier scalar searches missed it because they treated normalize as one global scale and never enumerated a hybrid that mixes two pair-partitions in a single pass. Reproducing the two-norm, per-lane divide matches the reference exactly and stabilizes the downstream PathIDs.
