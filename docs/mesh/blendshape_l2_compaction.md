# Blendshape compaction uses Euclidean (L2) delta magnitude

**Why it matters:** This was the dominant source of glb-wearable bundle size divergence. abgen-rs emitted more blendshape (morph) delta vertices than the reference, which lengthens `m_Shapes.vertices` and shifts every subsequent byte in the Mesh body, cascading into a large per-mesh diff and a slightly larger compressed bundle. The avatar heads and bodies with dense morph targets were the worst offenders. Crucially, the vertex stream layout, channel set, and data size were already byte-identical — the only structural difference was how many morph delta vertices survived compaction.

**How it works:** Unity's mesh importer drops a morph-delta vertex when the delta is negligible, and the keep/drop test is on the Euclidean (L2) magnitude of the delta vector, not on any single component. abgen-rs previously used a per-component threshold, which cannot reproduce the reference: the per-component metric overlaps the boundary (the importer keeps some vertices with smaller max-component deltas than ones it drops), so no per-component cutoff separates the two sets cleanly. The L2 magnitude separates them with a sharp, one-sided boundary at the importer's well-known blendshape import epsilon. The rule is: keep a morph-target vertex if the L2 magnitude of its position delta or its normal delta meets that epsilon; otherwise drop it. abgen-rs applies this single global epsilon to the delta vector magnitudes, which is content-agnostic and generalizes across meshes.
