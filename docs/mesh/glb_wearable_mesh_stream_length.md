# glb-wearable mesh vertex-stream length — already byte-exact

**Why it matters:** A set of glb-wearable bundles miss byte-identity by small on-disk sizes, and the suspected cause was the mesh vertex-stream length — that our channel-inclusion or interleaved-stride packing in `mesh_layout.rs` / `gltf.rs` was emitting a vertex/index stream of the wrong length, which would be a genuine structural bug.

**How it works:** This is largely a negative finding. The size figures in the parity report are LZ4-compressed bundle sizes, not uncompressed object sizes, so a small compressed delta does not imply any object changed length. Decompressing and comparing the actual CAB and `.resS` payloads shows every mesh's vertex/index stream and Mesh object header is exactly the reference length — zero meshes differ in size corpus-wide. The channel-inclusion rule (positions and normals always, normals recalculated when absent; tangents only when the source has TANGENT or the material has a normal map; colors only with COLOR_0; all TEXCOORDs; bones only with WEIGHTS_0 and JOINTS_0) and the interleaved stride/stream packing already reproduce the reference's exact stream length. The residual on-disk deltas are LZ4 reacting to bit-value differences in the CAB (mesh AABB / normal / tangent quantization and path-id ordering) at identical input length, and they collapse automatically as those bit-value axes close — there is no independent size fix in the mesh-layout code.

One adjacent, separable bug surfaced: Unity appends a `_N` de-duplication suffix to duplicate node/mesh-primitive GameObject names that abgen-rs leaves bare, which is a glTF node-naming rule in `gltf.rs`, not a vertex-stream issue. The lasting methodological point: any size-class triage must decompress and compare CAB/`.resS` lengths before attributing a delta to a structural cause, or bit-value diffs masquerade as size diffs.
