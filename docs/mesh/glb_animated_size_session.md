# glb-animated bundle size — multi-primitive collider merge

**Why it matters:** Animated GLB bundles were coming out larger than the reference, and the dominant recoverable cause was a collider built with the wrong object structure. When a glTF mesh has several primitives that share their entire vertex stream (same POSITION/NORMAL/TANGENT/TEXCOORD/COLOR/JOINTS/WEIGHTS accessors, differing only in indices and material), GLTFast builds a single Unity Mesh with multiple sub-meshes. abgen already merges these for renderer meshes, but its merge path explicitly bailed on collider nodes, splitting each extra primitive into its own GameObject/Transform/MeshFilter/MeshCollider/Mesh set. That produces many extra objects and duplicated collider mesh copies, inflating bundle size and skewing the parent Transform's child list.

**How it works:** The fix extends the shared-vertex-stream merge to collider nodes. A merged collider emits one merged Mesh plus a MeshFilter and a MeshCollider, and crucially no MeshRenderer (correct for renderers, wrong for colliders); collider meshes also drop their materials. The merged geometry invariants — shared vertex stream from the first primitive, concatenated index buffer, one sub-mesh per primitive each carrying the mesh-wide bounds — are identical to the renderer merge. The merge fires only when a node has multiple primitives with shared accessors and is a collider, and never on skinned or morph-target meshes, which keeps the skinned-renderer/baked-collider path authoritative for those.

The remaining glb-animated size deltas are irreducible: extra-Material structural differences (a separate known wall), texture `.resS` payload length from the BC7/crunch encoder, and PathID renumbering plus LZ4 framing noise over identical object sets. The Animation, AnimationClip, and mesh-vertex-stream classes contribute no recoverable structural size in this bucket.
