# Cross-bundle external texture references

**Why it matters:** Decentraland's converter splits an entity's content across several AssetBundles — one per glTF mesh, one per standalone image. When a mesh references an image whose bytes live in a sibling image bundle, Unity does not inline a local texture; it emits a cross-bundle pointer into the material. Inlining a local texture instead, or getting the external wiring wrong, breaks byte-identical output for the affected material, AssetBundle, and metadata objects.

**How it works:** For an image that resolves to a sibling bundle, the builder emits a `PPtr` whose file id is the slot of that sibling in the bundle's externals list (slot zero is always the shader, siblings start after it, deduplicated per sibling) and whose path id is the texture's id inside its own bundle. Each sibling external is recorded with a zeroed guid and the archive path of its CAB. The sibling CABs are appended to the bundle's dependency list and then sorted alphabetically (a prod-derived rule the Python reference omits), and the sibling bundle filenames are written into the metadata's dependencies array in encounter order. The cross-bundle pointers are also spliced into the material's preload run after the shader and local textures.

This is gated behind a hash-resolver callback: the full-scene driver supplies it, while the byte-parity test harness leaves it off so existing fixtures stay bit-identical. Some cross-bundle cases remain blocked by unrelated bugs (a default-material preload shortfall, and a glTF-vs-glb detection issue that misroutes `.gltf` content to the standalone-texture path, which never consults the resolver).
