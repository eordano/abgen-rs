# Cross-bundle texture pointers for plain .glb materials

**Why it matters:** A Material's texture slots must point at the Texture2D objects that hold the image data. When an image lives in a sibling content file that ships as its own standalone-texture bundle, Unity emits a cross-bundle pointer (a non-local FileID plus the standalone bundle's precomputed PathID). abgen-rs's resolver can produce that pointer, but only when it is told the source-file path and the entity's content map. On the windows reference set, the surviving Material diffs were materials whose texture pointers fell back to the local-pointer default because the resolver was never activated — the slots came out `(0, 0)` instead of the real cross-bundle pointer.

**How it works:** This was a measurement-harness gap, not a builder bug. The resolver already handled this case; it activates whenever a source file and a non-empty content map are supplied. The original measurement scripts only passed those arguments for `.gltf` and `_emote.glb` entities, suppressing the resolver for ordinary `.glb` entities even when their embedded glTF references an external image by URI. The fix is to always supply the source-file path and content map whenever the entity manifest is available, regardless of file extension — closing the Material class on windows (and, by the same platform-agnostic resolver path, on mac). Activating the resolver also surfaces residuals in other classes (AssetBundle preload-table ordering, TextAsset metadata version) for the same bundles; those are pre-existing and tracked separately.
