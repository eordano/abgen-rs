# Shader-slot position is not derivable from bundle content (negative result)

**Why it matters:** With the per-target majority default in place (FIRST on windows and mac, LAST on linux), a minority of bundles still want the opposite shader-slot position. Closing that residual statically would require a content-derivable rule that decides FIRST vs LAST per bundle. This document records an exhaustive search for such a rule and concludes that none exists.

**How it works:** This is a negative finding. A wide feature set was harvested from each bundle's glTF document, CID, asset GUID words, and material/mesh/texture shape — plus prod-observable object counts included only as a separability sanity check. An exhaustive single-, pair-, and triple-feature search, plus bit-level and modular splits over every numeric feature, was run against the prod-labeled slot direction. No rule cleared a useful accuracy bar; the best single feature barely beats the majority baseline and adding features yields an essentially flat, marginal gain. Even the prod-observable features (which a real builder could not use) fail to separate the population — proof the signal is not in the bundle at all.

The decisive evidence is cross-platform: joining each CID's slot direction across windows and mac shows a large fraction of CIDs flip direction between platforms. Same content, same CID, opposite decision. That is only possible if the slot is set by per-target editor state — Unity's native serializer encounter order, driven by InstanceIDs the editor hands out per build — rather than by anything fixed at upload time. Since Unity IPC and per-CID lookup tables are both off-limits, the only legitimate path to close the residual is emit-and-verify against a known prod hash, which is what `assetbundle_expect_hash.md` implements.
