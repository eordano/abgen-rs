# AddObjectToAsset sub-asset PathIDs are not derivable

**Why it matters:** When Unity attaches a sub-asset to a parent asset (the `AssetDatabase.AddObjectToAsset` path used for emote AnimatorControllers and their clips), each sub-asset gets a `localIdentifierInFile` (LFID / PathID). To produce byte-identical bundles, abgen-rs must reproduce those exact PathIDs. The hope was that the LFID is a pure function of `(parent guid, child name, child type)` that we could compute offline.

**How it works:** This is a negative finding. Running the same controlled probe twice against the same project — identical parent guids, child names, and child types — yields *different* sub-asset LFIDs every session. Only the well-known per-class main-asset IDs are stable (e.g. AnimatorController, Material, Texture2D); even prefab main objects vary. An exhaustive sweep of plausible byte-payload-and-hash combinations found zero matches. The conclusion is that Unity's generator carries hidden editor-session state (instance-id-derived or seeded by a session RNG) and only records the result into the asset YAML after the fact, so no formula of the inputs can predict it.

Consequently, "probe Unity and derive the formula" is dead. The remaining options are to override the LFID upstream at conversion time (a `ScriptedImporter` / external-object-remap change in the converter, producing deterministic IDs abgen-rs can mirror) or to accept the affected emote bundles as a known PathID-nondeterminism divergence rather than a tractable per-byte residual. No abgen-rs source changes follow from the probe itself.
