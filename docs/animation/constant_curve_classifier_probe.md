# Probe: where Mecanim's constant-curve classifier lives (negative finding)

**Why it matters:** When the converter bakes a humanoid AnimationClip it splits curves into a streamed group and a constant group, and matching that split exactly is required for byte-identical emote bundles. To reproduce the split, abgen-rs needs the rule the converter uses to decide which curves are constant. This probe set out to derive that rule by inspecting a real production bundle, without driving the Unity Editor.

**How it works:** Decoding a prod emote bundle yields the ground-truth verdict per binding. Two durable facts fell out: classification is per-binding-vector, not per-component (if any component varies, the whole binding stays streamed), and the serialized layout partitions all streamed bindings first, then all constant ones, each group preserving ingest order.

This is a negative finding on the rule itself. No classifier defined over the source glTF samples reproduces prod's verdict: bit-equality of source keyframes is neither necessary nor sufficient, and for rotation the boundary is inverted (constant bindings can carry large source deltas from quaternion sign flips while streamed bindings can be perfectly flat). The strongest discriminator is which bone a binding targets, not its values. This points to the classifier operating on a post-import representation abgen-rs does not mirror — most likely the humanoid muscle-space projection (possibly with quaternion shortest-path normalization or rest-pose subtraction). That projection runs only when a humanoid avatar is attached and lives in the native animation engine the converter drives, so the rule is not recoverable by inspecting glTF or by reflection. Until it is cracked, curve partitioning stays a pass-through.
