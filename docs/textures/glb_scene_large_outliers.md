# BC7 placeholder blocks in large glb bundles

**Why it matters:** The largest size outliers across glb-scene, glb-animated, and glb-wearable bundles all diverge in the streamed `.resS` texture payload. The uncompressed payload size matches the reference, but the compressed bundle is much larger because abgen-rs encodes real, high-entropy BC7 from the GLB's RGBA32 source while the reference ships a degenerate, highly compressible block. This is the dominant source of positive bytes-on-disk delta for textured glb bundles.

**How it works:** For these textures Unity does not actually run a real BC7 compressor. Instead it emits one fixed 16-byte BC7 mode-5 block (a uniform-anchor placeholder) and replicates that single block across every block of every mip level. A mip chain of one repeated 16-byte run compresses to almost nothing under LZ4HC, whereas real BC7 does not. The source images are genuinely multi-color, so this is Unity emitting placeholder bytes rather than encoding the content — the likely cause is the headless/no-GPU build context falling back to a default fill. A rare alternate placeholder block with a different endpoint pair shows up occasionally but has the same uniform-block-per-mip shape.

abgen-rs reproduces this by a per-Texture2D policy switch: instead of encoding real BC7, the placeholder path fills the entire mip chain with the fixed 16-byte block. This is applied to glb-bundle BC7 textures, where Unity always ships the placeholder. Runtime quality risk is nil because, by definition, abgen-rs is then shipping the same bytes as the reference.
