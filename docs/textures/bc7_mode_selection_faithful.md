# BC7 mode selection — matching bc7e faithfully

**Why it matters:** The converter compresses BC7 textures with Binomial's `bc7e.ispc` encoder (the open-source Intel/GameTechDev encoder, Apache-2.0; see NOTICES.md) at its `slow` profile. For each block the encoder picks one of the eight BC7 modes, and abgen must pick the same mode to have any chance at byte-identical output. abgen had accreted several empirical per-mode biases on top of the encoder's plain mode comparison, and those biases caused it to pick different modes than the reference — most visibly over-picking modes 4 and 6 where the reference used mode 5 (notably on fully-transparent tiles routed through the alpha path).

**How it works:** Upstream bc7e selects the winning mode with a plain strict-less-than error comparison and no per-mode bias; ties fall to the earlier-evaluated mode. abgen now reproduces that comparison directly — this is the default. The empirical biases it had accreted have all been removed: the mode-6 preference, the mode-1 low-variance penalty, mode-5 alpha/uniform special-casing, the transparent mode-5 `>>1` discount, the normal-map context gate, the mode-1 partition-shortlist early-cap/bonus, and the mode-1 ±1 endpoint perturbation. They were kept opt-in only because they tripped a few `parity_bytes` caps on mode-1 texture blocks; those fixtures and the gate have since been deleted, so nothing remains to keep them off.

When this change landed it lifted byte-identical textures on both platforms with no regression on any non-texture kind. It does slightly widen the LZ4 size gap on near-miss textures — that drift is the within-mode value gap below, not a mode-selection error.

That value gap is the real wall and is irreducible: even with correct mode selection, the endpoint/selector search inside a mode differs from bc7e by one or two quantization steps. The textures where abgen picks the reference's mode and produces the identical compressed size but a different bit pattern (smallest few bits) isolate it. The deltas trace to float-reduction order — AVX2 summation versus ISPC's instruction order produces ULP differences that change the final endpoint rounding. This is a numerical-precision difference, not an algorithmic divergence, and cannot be closed without reproducing ISPC's exact float operations.
