# Texture2D import flags already match Unity (negative finding)

**Why it matters:** A Texture2D object carries a cluster of import-flag fields in its header — readability, streaming-mipmap flags, lightmap format, color space, mipmap-limit handling, and the texture settings (filter mode, anisotropy, mip bias, wrap modes). If abgen-rs derived any of these differently from Unity's importer, the header bytes would diverge and break byte-identical output. The hypothesis was that some of the standalone-texture residual lives in these header flags rather than in the image data.

**How it works:** This is a negative finding — the import-flag cluster is already byte-exact against the reference for both standalone-texture and glb-embedded Texture2D objects. A flag flip would only change scalar header fields, so it could only ever explain *size-matched* bundles; in practice every size-matched standalone bundle has zero header-field differences, and the differing bytes always sit deep in the BC7 image-data region, never in the header at the front of the object. abgen-rs reproduces the flags correctly because the texture profile derives color space and lightmap format from a normal-map filename heuristic plus a per-material-slot color-space override, derives filter and wrap modes from the glTF sampler, derives readability from the streaming gate, and uses the DCL importer defaults for the remaining fields. The only way to move the standalone-texture row further is the BC7 image-data encoder, a separate workstream.
