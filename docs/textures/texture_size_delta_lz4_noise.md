# Tiny on-disk texture deltas are LZ4 noise over BC7 texel diffs (negative finding)

**Why it matters:** Many standalone-texture bundles differ from the converter reference by only a few bytes on disk. Such tiny deltas could mean two very different things: a real structural difference in the texture header or field lengths (recoverable with a converter fix), or compressed-size noise downstream of slightly different image content (irreducible here). They could also, in principle, point at the LZ4HC compressor itself diverging from liblz4. Distinguishing these is necessary to know whether there is any fix to make.

**How it works:** This is a negative finding — the small deltas are LZ4HC compressed-length noise produced by feeding the compressor slightly different BC7 texel bytes. When the raw decompressed bytes are compared, the texture envelope is byte-identical: the Texture2D header, the AssetBundle container, the preload table, the mip layout, `m_CompleteImageSize`, and the stream flags all match exactly. The only differing bytes are a handful of BC7 4×4 blocks inside the image data. Because the raw uncompressed size is identical for every small-delta bundle, none of them are structural. The compressor is also exonerated: whenever the raw input bytes are identical the on-disk bytes are identical too, confirming that abgen-rs's LZ4HC matches liblz4 byte-for-byte. The only way to shrink these deltas is to make abgen-rs's BC7 encoder produce the same blocks as the converter's bc7e output, a separate workstream. A few large textures do differ in raw length, but those belong to the streaming-gate/missing-content class, not the texture envelope.
