# Standalone-texture structural size is already correct (negative finding)

**Why it matters:** Most standalone-texture bundles still differ from the reference in final on-disk size, which threatens byte-identical output. A natural suspicion is that the texture's structural size is wrong — that abgen-rs disagrees with the reference on mip count, `m_CompleteImageSize`, the texture format, or where the image data is split between the inline payload and the `.resS` sidecar. If the byte count were wrong, no amount of texel-level work could ever close the gap, so getting the count right would have to come first.

**How it works:** This is a negative finding — the structural size is already exact. Across the corpus, abgen-rs matches the reference on every size-determining field: width/height, mip count, texture format, `m_CompleteImageSize`, readability, the inline-vs-`.resS` streaming decision, and the lengths of both the inline image data and the `.resS` sidecar. The uncompressed serialized payload length matches everywhere. The remaining on-disk size differences come entirely from the *content* of the BC7 image data: same dimensions, same mip chain, same byte count, but different texel bytes, which compress to a different LZ4 output length. A small residual also comes from the AssetBundle container's LZ4 block framing emitting different compressed bytes for an identical serialized payload. Neither is a size/mip/split problem; both are out of scope for size parity and belong to the BC7 encoder and container-framing workstreams.
