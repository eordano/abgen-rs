# Crunch RDO quality calibration for BC5 normal maps

**Why it matters:** Unity encodes BC5/DXT5Crunched normal-map textures (Texture2D, "Hx" magic) through crnlib's rate-distortion-optimizing CRN compressor at a specific quality level. The encoder's quality knob is the dominant lever on the compressed byte stream, so picking the wrong value makes our CRN output diverge from prod even when the source image and target dimensions match exactly — blocking byte-identical bundles.

**How it works:** crnlib's RDO trades bits against distortion as a function of `m_quality_level`. Sweeping that value across the BC5 corpus (restricted to textures where our dimensions already match prod, so the measurement isolates RDO behavior rather than resize residuals) traces a clean distortion-vs-quality parabola with a single minimum. abgen-rs sets the encoder to that minimizing quality in `src/builder.rs::texture` rather than crnlib's upstream default. Per-texture, a few small textures prefer a higher quality, but the project's discipline rule forbids per-asset tuning: the aggregate minimum is what ships, because the textures that "want" higher quality are small and contribute little absolute difference.

The other crnlib parameters are not tunable for parity reasons — they are either format-correctness requirements (the perceptual flag must be off for normal data, the alpha component selects the Y channel, the file type is CRN not DDS) or determinism requirements (single-threaded). The residual that remains after correct calibration is irreducible from our side: it comes from Unity's NPOT resize pipeline (most BC5 source images need resizing before encoding) and from possible drift between the vendored public-domain crunch and whatever revision Unity ships. Closing it would require an exact-match resize pipeline and/or porting Unity's specific crunch fork.
