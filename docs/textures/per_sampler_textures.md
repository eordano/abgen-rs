# Per-sampler Texture2D duplication

**Why it matters:** When one glTF image is referenced through more than one sampler, Unity emits a distinct Texture2D per (image, sampler) pair. abgen-rs cached textures by image index alone, so it coalesced them into a single object. That had three effects: prod's extra Texture2D objects had no counterpart, the wrong sampler's wrap/filter settings leaked into every material's slot, and — because one duplicated uncompressed texture was being streamed that prod kept inline — every later texture's stream-data offset was shifted, cascading into a large run of false diffs.

**How it works:** The texture cache and naming key on the (image, sampler) pair, not the image alone. The first sampler seen for an image keeps the plain `image_N` name; later samplers get an `image_N_samplerM` suffix where `M` is the glTF sampler index (not a sequential counter). Filter and wrap settings are looked up per sampler, which fixes the silent wrap-mode leak; a per-image fallback preserves the single-sampler path unchanged.

The inline-versus-streamed rule is the other half. Unity keeps the in-glb uncompressed Texture2D inline and readable exactly when its image is referenced through more than one distinct sampler; only the BC7 variants go to the external stream. Image raw size is not the predictor — large single-sampler images still stream normally. abgen-rs precomputes the set of distinct samplers per image, marks the multi-sampler uncompressed textures to stay inline, and only wipes inline image data for entries that actually landed in the external stream.
