# Texture streaming gate is `m_IsReadable`

**Why it matters:** Whether a Texture2D's pixels are streamed into the `.resS` sidecar or kept inline determines the byte offsets of every subsequent streamed texture in the same bundle. abgen-rs streamed any texture with non-empty image data, ignoring readability. When the reference kept one in-glb uncompressed texture inline that abgen-rs streamed, every later streamed texture in that bundle landed at the wrong `m_StreamData.offset` — a cascading shift equal to the inline texture's size, corrupting the whole bundle's parity.

**How it works:** the converter streams a texture into `.resS` if and only if `m_IsReadable` is false; no production texture is ever both streamed and readable. So the streaming gate is exactly `do_stream = !m_IsReadable`. The remaining question is when an in-glb uncompressed texture becomes readable: this happens when the glTF loader creates the texture at runtime via an image-load path (rather than through the `TextureImporter`), which yields a readable texture and keeps it inline. Empirically this occurs for uncompressed textures at exactly the editor's default desktop max dimension; textures that exceed that limit instead take the importer path, come back non-readable, and stream normally. The narrow predicate "uncompressed and at exactly the max dimension marks itself readable/inline, everything else streams" matches every observed case. This is a thin sample, so the full rule the converter follows is likely richer, but the predicate holds across the corpus without false positives.
