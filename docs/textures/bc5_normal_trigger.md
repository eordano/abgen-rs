# BC5 normal-map format: when Unity selects it, and why we can't fully match it

**Why it matters:** For byte-identical output, abgen-rs must emit each streamed Texture2D in the exact format Unity's TextureImporter chose. A subset of glTF images come out as BC5 with the normal-map flag set instead of the usual BC7. If abgen-rs guesses the wrong format, the texture's metadata and pixel payload both diverge from prod.

**How it works:** The trigger is derivable from the glTF document alone, with no Unity-internal hints. An image is encoded as BC5 (with the normal-map flag) exactly when it is bound to a `material.normalTexture` slot in at least one material AND also bound to a base-color or emissive slot in at least one material. In other words, an sRGB color image that is simultaneously reused as a normal-map input. This is the complement of the single-purpose normal-map set: a pure normal map stays in the normal-map path, while this dual-use intersection is what flips to BC5.

The encoder side is a known, irreducible gap. Unity does not ship these as raw BC5: it Crunch-compresses them (the streamed payload carries the Crunch header magic and is several times smaller than a raw BC5 mip chain). The same TextureImporter ships some other formats raw — the Crunch decision is made per-format. Shipping raw BC5 from abgen-rs would correct only the few format/flag metadata bytes; the payload stays a full-size raw chain that mismatches prod's compressed length, and the block bytes diverge from Crunch's output regardless. The only way to truly close the gap is to emit Crunch-compressed BC5, which requires porting or binding Crunch's RDO encoder — a large effort for a small slice of the corpus. The trigger rule is recorded here for reuse; the encoder change is deferred until a Crunch BC5 encoder is in scope.
