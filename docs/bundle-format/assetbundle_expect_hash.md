# Opt-in emit-and-verify for the shader-slot ambiguity (`BuildOpts.expect_hash`)

**Why it matters:** The position of the URP shader external within each material run (FIRST vs LAST) is set by Unity's per-target editor state and cannot be predicted from bundle content (see `assetbundle_shader_slot_rule_v2.md`). The per-target majority default gets most bundles right but leaves a minority wrong, and no static rule can close that gap. For parity-replay pipelines that already hold the prod bundle, the remaining shader-slot residual is recoverable.

**How it works:** `BuildOpts.expect_hash` takes the hex SHA-256 of the prod bundle. When supplied, the builder builds once with the per-target default slot, hashes the result, and returns immediately on a match. On a miss it rebuilds with the opposite slot and returns that if it matches. If neither matches, it returns the **first** build with a warning on stderr — returning the second build instead would flip bundles whose slot was already correct (their residual lay in another class) to the wrong slot, hurting parity. The decision is a single bit derived from the hash-match outcome, not a recorded per-CID direction, so it stays within the no-lookup-table policy.

When no `expect_hash` is given the path is unchanged: a single build with the per-target default, zero overhead. The cost on opt-in builds is up to one extra build per bundle that misses on the first slot, paid only in parity-replay. The mechanism also finished wiring the target-aware `ExternalsPosition::for_target` default into the glb build call site, which had still been defaulting to LAST everywhere. The CLI exposes `--expect-hash` and `--expect-hash-file`; the standalone-texture path is unaffected since those bundles carry no externals.
