# AssetBundle shader-slot position on mac (StandaloneOSX)

**Why it matters:** AssetBundle typetree objects diverged from prod on the mac corpus because the URP shader external sat in the wrong position within each material run. abgen-rs had inherited the LAST default from linux, but mac prod bundles overwhelmingly place the shader external FIRST. Every misplaced slot rewrites the `m_PreloadTable` file/path IDs, blocking byte-identical AssetBundle output.

**How it works:** The shader-slot direction is set by Unity's editor-side InstanceID assignment for the shader bundle at build time — per-target state that abgen-rs cannot reproduce without invoking Unity. The robust closed-form approximation is the per-target majority slot. On mac the majority of shader-carrying bundles emit FIRST (matching windows), so `ExternalsPosition::for_target` was extended to return `First` for mac as well as windows, with everything else staying `Last`. The change is mirrored in the Python reference. The source-extension-aware container-key rule from the windows work already covers mac's glTF-sourced bundles, so no additional fix was needed there.

A minority of mac bundles still want LAST under the new FIRST default. That residual is irreducible by any static rule — the per-CID direction is not a function of content (see `assetbundle_shader_slot_rule_v2.md`). The only legitimate way to close it is emit-and-verify when a prod hash is available (see `assetbundle_expect_hash.md`). A handful of remaining AssetBundle field-diffs are knock-on effects of cross-bundle external plumbing in textures and text assets, out of class for the shader-slot fix.
