# PathID collisions on duplicate sibling node names

**Why it matters:** Bundles containing glTF nodes that share a name (same-named sibling meshes, all-unnamed nodes that collapse to a default name, deep chains of unnamed nodes, per-glyph mesh primitives, repeated skeleton bones) failed to build with a hard `PathID collision` error. abgen-rs derives a node's PathID by hashing a recycle-name string, so two siblings with the same name produced the same PathID and the build aborted. These were dropped from the corpus entirely rather than being compared.

**How it works:** This is partly an irreducible divergence. The converter's scriptable build pipeline assigns distinct PathIDs to same-named siblings using the `PrefabPackedIdentifiers` strategy, which keys off the internal object-serialization identity (a hash of the object identifier), not the gameObject name. abgen-rs only sees the recycle-name string, so it cannot reproduce that discriminator and cannot match the reference's PathIDs for duplicate siblings.

What abgen-rs does instead is detect the collision at role-insertion time: when a recycle-name key is already claimed by a different PathID, it disambiguates by appending a duplicate suffix and re-claiming, so the build succeeds and produces a loadable bundle. Re-setting the same role at the same PathID stays a no-op (Transforms are legitimately set twice — placeholder then full tree). These disambiguated bundles do not match the reference on the duplicate nodes' PathIDs, but they now build and contribute a small residual instead of failing outright. Matching the reference exactly would require reverse-engineering the `PrefabPackedIdentifiers` discriminator, which uses a different hashing scheme than abgen-rs.

A separate failure in the same investigation — a glTF requiring `KHR_draco_mesh_compression` — is handled by bailing early with a clean unsupported-extension error, converting a downstream panic into a categorized build error. abgen-rs has no Draco decoder, and even with one the decoded mesh attributes would have to match the converter's GLTFast Draco output for byte parity.
