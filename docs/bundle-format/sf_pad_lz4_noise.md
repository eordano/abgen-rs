# Why stripping the trailing SF pad can make a fixture's parity metric look worse

**Why it matters:** Removing the trailing align-to-16 padding after the last object in a SerializedFile is the correct behavior — Unity does the same strip. But for one fixture the parity metric (`bits_different`) went slightly up after the change, which looks like a regression. Understanding why this is a false alarm matters so future cap movements are not misread as the produced content getting worse.

**How it works:** The parity metric combines two terms: a content term (XOR-popcount over the common prefix of the two byte streams) and a length-penalty term proportional to the absolute size difference. Removing highly-compressible trailing zeros shrinks the LZ4HC output by a couple of bytes. Before the strip, the compressed length happened to coincide exactly with prod's by luck, so the length penalty was zero; after the strip we land just below prod's compressed length, paying a small length penalty even though the content term improved. The net is a tiny metric increase driven entirely by LZ4 compressed-length noise, not by worse content.

**The durable insight:** compressed length is noisy, so a small move in a fixture's parity cap is not by itself evidence of a content regression. The `parity_decompose` example separates the content (common-prefix XOR) term from the length-penalty term, so the question "did the bytes get worse, or did the length penalty shift?" can be answered directly. This is a negative finding: no code change was warranted, and the architectural strip is correct.
