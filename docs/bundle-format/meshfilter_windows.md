# Scene-name and .gltf wrap rules for recycle paths

**Why it matters:** Scene-object PathIDs (GameObject, Transform, MeshFilter, MeshRenderer) are derived from a recycle-name path whose prefix includes the scene name and the scene-wrapping layers. If that prefix differs from Unity's, every scene object in the bundle gets a different PathID and the bytes diverge — even though the MeshFilter pipeline itself is correct. On the windows reference set the MeshFilter class was already at full paired-PID parity; the only residuals were unpaired objects in two bundles, caused entirely by recycle-prefix drift upstream in the scene layer.

**How it works:** Two upstream rules close the gap. First, the scene name in the recycle prefix must be taken from the glTF `scene.name` field rather than hard-coded; when a scene has no name, Unity uses an empty string, which collapses to a different prefix than the hard-coded literal. Second, Unity always emits an extra wrap GameObject layer for `.gltf` (text) source files, even when the scene has a single root node and no animations — abgen-rs previously only added the wrap for multi-root or animated scenes, so single-root `.gltf` sources came out a layer short.

Both rules change the root recycle prefix and therefore ripple through the PathIDs of every scene object in the affected bundle, so they belong in the scene-rebuild layer rather than as MeshFilter-class edits — there is no MeshFilter-internal bug. With both rules landed, the affected bundles match Unity at the MeshFilter (and MeshRenderer / SkinnedMeshRenderer / MeshCollider) layer.
