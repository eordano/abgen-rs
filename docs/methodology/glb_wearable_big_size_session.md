# glb-wearable large size outliers — BC7 texel and LZ4 compression noise

**Why it matters:** Some glb-wearable bundles diverge from the reference by large on-disk sizes — far more than rounding noise — which looks like it could be a structural over- or under-emission: an extra or missing mesh, a missing texture, a wrong blendshape stream, or a different mip/`.resS` split. Any of those would be a real builder bug worth fixing.

**How it works:** This is a negative finding. In every outlier the object count is identical, the external-dependency set is identical, and the decompressed CAB and its `.resS` texture stream have byte-for-byte identical *lengths* on both sides — no extra or missing object, no stream-length difference. The entire size gap is LZ4 compressing differing-but-equal-length content, overwhelmingly the BC7 texel bytes in `.resS`: our pure-Rust BC7 encoder produces a different byte pattern than the reference for the same image, and identical-length-but-different texels compress to a different size. A header parse confirms the uncompressed block table (block count and sizes) is identical between sides; only the compressed payload and file length differ. The `.resS` stream holds only texture pixel blobs, so a `.resS` byte diff is necessarily a texel diff, never a mesh or blendshape issue; the CAB adds at most a handful of float-precision mesh-vertex bytes. The sign of the delta just reflects whether our texels happen to compress better or worse than the reference's. This is irreducible without bit-exact replication of bc7e (the open-source Intel/GameTechDev encoder the converter invokes), which is out of scope; the honest parity signal here is object-set plus decompressed-length parity, not compressed bundle size.
