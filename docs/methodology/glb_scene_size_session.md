# glb-scene structural size parity — negative finding

**Why it matters:** glb-scene bundles cluster around very small on-disk size deltas, and the hypothesis was that this cluster represented a single systematic *structural* divergence — a variable-length string emitted one character different, a varint count, or 4/8/16-byte alignment padding at an object boundary. A structural root cause would be fixable and would close many bundles together, so it was worth ruling in or out before chasing bit-value tails.

**How it works:** The hypothesis is refuted. For every size-mismatched bundle, decompressing both ours and the reference to their raw UnityFS dir-nodes shows the decompressed CAB is byte-identical in length; the on-disk delta carries no structural insert or delete. The small deltas are LZ4HC reacting to content (bit-value) differences, not length differences — which follows necessarily from the fact that our LZ4HC encoder is byte-exact with Unity's, so a same-decompressed-size pair that differs on disk must be a content reaction. A worked example localizes the change to `AssetBundle.m_PreloadTable` PPtr ordering: the material PPtr and an external-shader PPtr are transposed in the preload list, the recompressed bytes shift the on-disk size, and the decompressed payload stays the same length. The rest split between this preload transposition and `Transform` sign-bit / material-color float-LSB flips — all same-decompressed-size, all already-documented bit-value walls. The few larger deltas are real BC7 content drift on high-mip textures, a separate axis. The glb-scene structural size question is closed-negative.
