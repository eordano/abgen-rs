# glb-scene small size-class — re-confirmed negative at full scale

**Why it matters:** A large group of glb-scene bundles miss byte-identity by only a handful of bytes of on-disk size. The open question was whether this small size-class hides a systematic *structural* defect — a name written one character off, a varint count off-by-one, or an alignment pad inserted at an object boundary — that, if fixed, would close many bundles at once.

**How it works:** This is a negative finding. Decompressing every bundle in the class to its raw UnityFS dir-nodes (the CAB plus any `.resS`) shows the decompressed node set and every node's length are identical to the reference, with no structural insert or delete anywhere. The small on-disk deltas are pure LZ4HC compressed-frame-length noise: where the content differs, it is an in-place same-length byte swap, and the compressor simply emits a frame a few bytes longer or shorter in reaction. The underlying byte differences are already-documented bit-value walls — chiefly `AssetBundle.m_PreloadTable` PPtr ordering and the external-slot transposition that cascades into Material `m_FileID` swaps, both governed by Unity's build-time InstanceID allocation and not recoverable clean-room — plus a small remainder of real mesh/BC7 content drift in `.resS`. Since LZ4HC framing is already byte-exact with Unity's, a same-decompressed-size pair that differs on disk is definitionally the compressor reacting to content, not to length. There is no structural component to fix; the class should stay closed.
